# S1 SecOps Forge

**Discover, deploy, and share community-built apps for SentinelOne.**

S1 SecOps Forge is a community app store for SentinelOne SecOps apps: deployers, SDL solutions, skills, MCP servers, and integrations. It is a static site backed by a curated manifest and enriched with live GitHub metadata (stars, description, latest release, last updated, license).

> [!WARNING]
> **Community-supported. Read this first.**

## How it works

The manifest is the source of truth, and the volatile data comes from GitHub at build time so the published page never calls the GitHub API from a visitor's browser (which would hit the 60 requests/hour unauthenticated limit).

```
apps.json            you edit this (curated list, one entry per app)
   |
   v
scripts/enrich.mjs   GitHub Action runs it on a schedule
   |
   v
catalogue.json       generated: manifest + live stars/desc/version/etc.
   |
   v
index.html           static page renders cards from catalogue.json
```

If `catalogue.json` is missing (for example before the first Action run), `index.html` falls back to reading `apps.json` directly and enriches from the GitHub API in the browser, cached in `localStorage` for 6 hours.

## Add an app

Two ways:

1. **Pull request (preferred).** Add one entry to the `apps` array in [`apps.json`](apps.json), following the [schema](schema/app.schema.json). Run `node scripts/validate.mjs` locally, then open a PR. CI validates the entry and confirms the repo is public.
2. **Submission form.** Open a [new issue](../../issues/new?template=submit-app.yml) using the "Submit an app" template. A maintainer converts it to a PR.

Keep `tagline` short, or leave it empty to use the repo's own GitHub description. Do not hardcode stars, versions, or descriptions; those are pulled live.

### Manifest entry example

```json
{
  "id": "s1-ueba-deployer",
  "name": "S1 UEBA Deployer",
  "tagline": "One-shot UEBA anomaly detection deployment for SDL",
  "repo": "pmoses-s1/s1-ueba-deployer",
  "category": "Deployers & Solutions",
  "type": "deployer",
  "s1_products": ["Singularity Data Lake"],
  "author": "pmoses-s1",
  "docs_url": "",
  "featured": false
}
```

Categories: `Deployers & Solutions`, `SOC & DFIR`, `Data & AI-SIEM`, `Integrations & Tooling`.
Types: `sdl-solution`, `deployer`, `mcp`, `skill`, `integration`, `tooling`.

## Run locally

```bash
# generate catalogue.json from the manifest (optional; the page also works without it)
GITHUB_TOKEN=<your_pat> node scripts/enrich.mjs

# validate the manifest
node scripts/validate.mjs

# serve the folder
python3 -m http.server 8080
# open http://localhost:8080
```

A token is optional but recommended for `enrich.mjs`. Without one you get the 60 requests/hour unauthenticated limit, which still covers a small manifest.

## Publish on GitHub Pages

1. Push this folder to a public repo.
2. In **Settings > Pages**, set **Source: Deploy from a branch**, branch `main`, folder `/ (root)`.
3. The **Enrich catalogue** Action refreshes `catalogue.json` hourly and on every manifest change. Each refresh commit triggers a Pages rebuild, so the live data stays current with no browser API calls.

Set `CATALOGUE_REPO` near the top of the `<script>` in `index.html` to your repo (`owner/name`) so the "Add your app" links point to the right place.

## Files

```
apps.json                     curated manifest (edit this)
catalogue.json                generated by enrich.mjs (do not edit by hand)
index.html                    the catalogue page
schema/app.schema.json        manifest entry schema
scripts/enrich.mjs            manifest + GitHub -> catalogue.json
scripts/validate.mjs          manifest validation for CI
.github/workflows/enrich.yml  scheduled enrichment
.github/workflows/validate.yml PR validation
.github/ISSUE_TEMPLATE/submit-app.yml  submission form
DISCLAIMER.md                 full community-support disclaimer
CONTRIBUTING.md               how to contribute
```
